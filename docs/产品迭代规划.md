> 版本：v1.2.0 起草 | 日期：2026-07-11
> 适用产品：道体·玄盾 活性防护 LLM 防火墙
> 当前版本：v1.2.0（核心引擎 + 桌面端 + Python SDK 已完成）

## 一、迭代背景与现状分析

### 1.1 v1.1.0 已交付能力

| 模块 | 能力 | 关键文件 |
|------|------|----------|
| 核心引擎 | 观察→学习→自动切换的活性防护 | `src/daoti_xuandun/reject_gate.py`、`xuandun.py` |
| 攻击样本库 | 200+ 条覆盖 OWASP LLM Top 10 六大类 | `desktop/xuandun-desktop/attack_samples.py` |
| 模拟测试 | 拦截率/误报率/漏报率报告 | `desktop/xuandun-desktop/simulation.py` |
| 桌面端 | Tauri 三端应用（Win/macOS/Linux） | `desktop/xuandun-desktop/src/` |
| Python SDK | FastAPI 集成 | `src/daoti_xuandun/integrations/fastapi.py` |
| 审计链 | SQLite + 哈希链日志防篡改 | `desktop/xuandun-desktop/src-tauri/src/db.rs` |

### 1.2 v1.1.0 关键缺口（本轮迭代要解决）

通过对现有代码的审查，识别出三个影响产品商业化落地的关键缺口：

1. **Dashboard 可视化深度不足**
   - `Dashboard.tsx` 的趋势数据仅保存在 React `useState` 中（`history.slice(-30)`），**应用重启即丢失**，无法做跨会话趋势分析。
   - SQLite `logs` 表（`db.rs` 第 44-56 行）只记录 `trust_level`、`reject_stage`，**未记录攻击分类**（如 direct_prompt_injection / jailbreak），无法做攻击类型分布分析。
   - 当前轮询间隔 2 秒（`setInterval(fetchStatus, 2000)`），仅展示当前会话的增量数据，无历史对比能力。

2. **无安全报告输出能力**
   - 当前仅在 `commands.rs` 第 110-114 行通过 `app.notification()` 推送桌面通知，**无周报/月报生成机制**。
   - 模拟测试报告（`SimulationReport` 接口）只在前端展示，**无导出能力**。
   - 无定时任务调度器，运维人员无法自动化获取安全态势。

3. **无企业级告警与集成通道**
   - 告警通道单一（仅桌面通知），**无法对接钉钉/飞书/邮件/Webhook/SIEM**。
   - 企业运维场景下，安全事件无法进入既有 SOC/SIEM 工作流，玄盾成为"信息孤岛"。

### 1.3 迭代目标

本轮迭代围绕"**让玄盾从可用工具升级为企业级安全运营组件**"这一主线，分三个方向推进：

| 方向 | 核心价值 | 目标版本 |
|------|----------|----------|
| 方向一：Dashboard 可视化增强 | 让安全态势可见、可比、可追溯 | v1.2.0 |
| 方向二：安全报告定期推送 | 让安全运营可汇报、可留档、可审计 | v1.3.0 |
| 方向三：企业系统集成 | 让玄盾融入企业 SOC/SIEM 工作流 | v1.4.0 |

## 二、方向一：Dashboard 可视化增强

### 2.1 目标价值

**解决的核心问题**：当前 Dashboard 是"实时快照"而非"态势分析工具"——数据重启即丢、无时间维度对比、无攻击分类视图，运维人员无法回答"上周攻击趋势如何""哪类攻击最多""和上月相比防护效果是否退化"等关键问题。

**交付价值**：
- 运维人员可基于历史趋势判断防护效果演化
- 攻击类型分布帮助识别当前面临的主要威胁
- 实时流量监控辅助容量规划与异常发现
- 历史对比支撑安全决策（如是否需要切换防御层级）

### 2.2 功能详述

#### 2.2.1 趋势分析（时间维度的拦截趋势图）

- **多时间粒度切换**：支持 1 小时 / 24 小时 / 7 天 / 30 天四个粒度的趋势查看
- **双指标曲线**：在同一图表中展示"请求量"与"拦截量"随时间变化，支持切换为"拦截率"视图
- **数据持久化**：趋势数据基于 SQLite 持久化，应用重启后历史趋势可恢复
- **交互能力**：支持鼠标拖选时间范围放大、悬停查看具体数值

#### 2.2.2 攻击类型分布（饼图 / 雷达图）

- **饼图视图**：按 OWASP LLM Top 10 六大类（direct_prompt_injection / indirect_prompt_injection / jailbreak / encoding_obfuscation / agent_attack / data_leakage）展示占比
- **雷达图视图**：多维度展示各类攻击的拦截情况，直观识别"哪类攻击防御薄弱"
- **联动筛选**：点击某类攻击，下方的拦截记录表格自动筛选该类攻击
- **时间范围联动**：攻击类型分布随顶部时间选择器联动

#### 2.2.3 实时流量监控

- **实时 QPS 曲线**：秒级粒度的请求量曲线，支持告警阈值线标注
- **当前会话摘要**：实时展示当前 1 分钟 / 5 分钟 / 15 分钟的 QPS、拦截率、平均延迟
- **异常突增提示**：当请求量或拦截量较基线突增 N 倍时，Dashboard 顶部出现告警条
- **流量来源 TOP**：按 session_id 维度展示流量来源 TOP 10（为后续多租户铺垫）

#### 2.2.4 历史数据对比

- **周期对比**：支持"本周 vs 上周""本月 vs 上月"的拦截量/拦截率对比柱状图
- **模拟测试历史**：记录每次模拟测试的拦截率/误报率/漏报率，展示防护能力随时间演化曲线
- **基准对比**：将当前拦截率与"内置基准测试结果"（100% 拒绝率）对比，提示防护退化

### 2.3 技术方案

#### 2.3.1 数据层改造（关键前置）

**问题**：当前 `logs` 表缺少攻击分类字段，无法支撑攻击类型分布分析。

**改造点 1：扩展 logs 表 schema**

文件：`desktop/xuandun-desktop/src-tauri/src/db.rs`（第 44-56 行 logs 表定义）

新增字段：
```sql
ALTER TABLE logs ADD COLUMN attack_category TEXT;      -- 攻击分类 key
ALTER TABLE logs ADD COLUMN latency_ms REAL;           -- 检测延迟
ALTER TABLE logs ADD COLUMN domain_distance REAL;      -- 域距离（用于分布分析）
```

在 `db.rs` 的 `init_tables` 中增加迁移逻辑（参考现有 `hash_version` 迁移模式，第 78-97 行）。

**改造点 2：新增时序聚合表**

为避免大数据量下的趋势查询性能问题，新增预聚合表：
```sql
CREATE TABLE IF NOT EXISTS stats_hourly (
    hour TEXT PRIMARY KEY,           -- '2026-07-11T14:00:00Z'
    total_requests INTEGER NOT NULL,
    total_blocked INTEGER NOT NULL,
    total_allowed INTEGER NOT NULL,
    avg_latency_ms REAL,
    category_breakdown TEXT,         -- JSON: {"jailbreak": 12, "encoding": 5}
    block_rate REAL,
    updated_at TEXT
);

CREATE TABLE IF NOT EXISTS stats_daily (
    day TEXT PRIMARY KEY,            -- '2026-07-11'
    total_requests INTEGER,
    total_blocked INTEGER,
    category_breakdown TEXT,
    simulation_results TEXT,         -- JSON: 当日模拟测试摘要
    updated_at TEXT
);
```

**改造点 3：写入侧改造**

- `commands.rs` 的 `protect` 命令（第 70-143 行）：调用引擎后，从 `reject_stage` 推断 `attack_category`（需引擎返回该字段，见下方 Flask 改造），写入 logs 表。
- 新增后台聚合任务：每小时将 logs 表增量聚合到 `stats_hourly`，每日聚合到 `stats_daily`。

**改造点 4：Flask 引擎返回攻击分类**

文件：`desktop/xuandun-desktop/engine_flask.py`（第 282-330 行 protect 端点）

在响应中新增 `attack_category` 字段：
```python
response = {
    "allowed": result.allowed,
    "trust_level": ...,
    "reject_stage": result.reject_stage,
    "attack_category": getattr(result, "attack_category", None),  # 新增
    "domain_distance": result.domain_distance,
    "timing_distance": result.timing_distance,
    "latency_ms": round(lat, 2),  # 新增
}
```

> 注意：核心引擎 `reject_gate.py` 的 `protect` 结果对象需扩展 `attack_category` 属性。该字段可通过"拦截阶段 → 攻击分类"的映射表推断（如 `reject_stage=luoshu_distance` + `distance > threshold` → 归类为对应攻击类型），或由引擎在检测时直接标注。此改造涉及核心算法层，需与算法负责人确认实现方式。

#### 2.3.2 后端 API 扩展

文件：`desktop/xuandun-desktop/src-tauri/src/commands.rs` + `desktop/xuandun-desktop/engine_flask.py`

#### 2.3.3 前端改造

文件：`desktop/xuandun-desktop/src/pages/Dashboard.tsx`

**现状**：当前 Dashboard 使用 `recharts`（已在 `package.json` 依赖中），趋势数据用 `useState` 保存最近 30 个点（第 42-46 行）。

**改造方案**：

1. **新增时间选择器组件**：顶部增加 1H / 24H / 7D / 30D 切换按钮，调用 `get_trend_stats`
2. **趋势图改造**：`AreaChart` 数据源从内存 `history` 改为后端 `get_trend_stats` 返回的持久化数据；保留实时增量更新能力（2 秒轮询追加最新点）

4. **实时监控区**：独立卡片展示实时 QPS 曲线（秒级），数据来自 `get_realtime_metrics`
5. **历史对比区**：新增"本周 vs 上周"对比柱状图，数据来自 `get_comparison_stats`
6. **模拟测试历史曲线**：新增折线图展示历次模拟测试的拦截率/误报率演化

**依赖说明**：`recharts@^3.8.1` 已在依赖中，PieChart/RadarChart 均已支持，无需新增前端依赖。

#### 2.3.4 性能考量

## 三、方向二：安全报告定期推送

### 3.1 目标价值

**解决的核心问题**：企业安全运营要求"安全态势可汇报、可留档、可审计"。当前玄盾的检测结果只在桌面端实时展示，运维人员无法向上级汇报"本周防护效果如何"，无法留档"本月安全态势报告"，无法满足等保/合规审计要求。

**交付价值**：
- 自动生成周报/月报，降低运维汇报成本
- 多格式导出（PDF/HTML/Markdown）适配不同场景
- 定时调度让报告"自动送达"，无需人工干预
- 报告模板标准化，确保汇报内容完整规范

### 3.2 功能详述

#### 3.2.1 周报 / 月报自动生成

- **报告类型**：周报（覆盖最近 7 天）、月报（覆盖最近 30 天）、自定义周期报告
- **生成触发方式**：
  - 定时自动生成（如每周一 9:00 生成上周报告）
  - 手动一键生成（Settings 页面新增"立即生成报告"按钮）
  - 事件触发生成（如检测到高危攻击爆发时生成专项报告）
- **报告归档**：生成的报告自动归档到 SQLite，保留历史报告列表，支持查看和重新导出

#### 3.2.2 报告模板设计

报告包含以下章节：

**1. 概要摘要**
- 报告周期、生成时间、引擎版本、当前防御层级
- 核心指标：总请求数、拦截数、拦截率、误报率（基于模拟测试）、漏报率
- 环比变化：与上一周期对比的增减百分比

**2. 拦截统计**
- 按日/按时的拦截量趋势图（嵌入图表为图片）
- 拦截率波动区间（最高/最低/平均）
- 高峰时段识别（拦截量 TOP 3 时段）

**3. 攻击趋势分析**
- 六大攻击类型的占比变化（与上期对比）
- 新增攻击模式识别（首次出现的攻击分类）
- 攻击强度趋势（日均攻击量变化）

**4. Top 攻击类型详情**
- TOP 5 攻击分类的拦截量、占比、代表性样本（脱敏后的 text_preview）
- 每类攻击的拦截阶段分布（reject_stage 统计）

**5. 防护效果评估**
- 模拟测试结果（如周期内执行过模拟测试）
- 与内置基准（100% 拒绝率）的差距分析
- 误报风险评估（基于良性样本测试）

**6. 建议措施**
- 基于数据的自动建议（如"越狱攻击占比上升 30%，建议提升至 STRICT 层级"）
- 学习进度建议（如"观察模式已积累 N 条，建议切换保护模式"）
- 阈值调整建议（基于误报/漏报数据）

**7. 审计信息**
- 报告生成时间、数据完整性校验（哈希链验证结果）
- 报告数据来源范围（时间区间、样本数）

#### 3.2.3 导出格式

- **PDF**：适合正式汇报，含图表渲染、分页、页眉页脚（含玄盾 logo 和机密标注）
- **HTML**：适合内部分享，保留交互能力（可折叠章节、悬停查看详情）
- **Markdown**：适合接入企业 Wiki/Confluence，纯文本便于版本管理

#### 3.2.4 定时任务调度

- **调度配置 UI**：Settings 页面新增"报告调度"配置区
  - 报告类型（周报/月报）
  - 生成时间（如每周一 9:00）
  - 导出格式（可多选）
  - 推送渠道（关联方向三的告警通道，如生成后发送到飞书/邮件）
  - 启用/停用开关
- **调度执行**：后台定时任务，应用启动时加载调度配置，到点自动生成并归档
- **失败重试**：生成失败时记录日志并重试 3 次，仍失败则桌面通知提醒

### 3.3 技术方案

#### 3.3.1 报告生成引擎

**技术选型**：Python 侧生成报告（复用 Flask 引擎进程），原因：
- 已有 `simulation.py` 的报告生成经验
- Python 的报告/图表生态成熟（Jinja2 模板 + matplotlib 图表）
- 避免在 Rust 侧引入重型模板引擎

**新增文件**：`desktop/xuandun-desktop/report_generator.py`

```python
class ReportGenerator:
    def __init__(self, db_path: str, shield):
        self.db_path = db_path
        self.shield = shield

    def generate(self, report_type: str, period_start: str, period_end: str) -> dict:
        """生成报告，返回报告数据结构。"""
        # 1. 从 SQLite 聚合数据（复用方向一的 stats_daily 表）
        stats = self._aggregate_stats(period_start, period_end)
        # 2. 生成图表（matplotlib 保存为 base64 PNG）
        charts = self._render_charts(stats)
        # 3. 渲染模板
        report = self._render_template(stats, charts, report_type)
        # 4. 归档到 DB
        self._archive(report)
        return report

    def export_pdf(self, report: dict) -> bytes: ...
    def export_html(self, report: dict) -> str: ...
    def export_markdown(self, report: dict) -> str: ...
```

**模板引擎**：使用 Jinja2（需加入 `pyproject.toml` 依赖），三套模板：
- `templates/report.html.j2` → HTML 输出
- `templates/report.md.j2` → Markdown 输出
- HTML → PDF 转换使用 `weasyprint` 或 `pdfkit`（需评估打包体积）

#### 3.3.2 图表渲染

**方案**：使用 matplotlib 生成图表 PNG，base64 嵌入 HTML/Markdown。

图表清单：
- 拦截趋势折线图（双轴：请求量 + 拦截率）
- 攻击类型分布饼图
- 每日拦截量柱状图
- 模拟测试历史曲线

> 注意：matplotlib 需在 Nuitka 编译时正确处理，参考 `build_engine.py` 现有的依赖打包方式。

#### 3.3.4 报告归档

扩展 SQLite schema（`db.rs`）：
```sql
CREATE TABLE IF NOT EXISTS reports (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    generated_at TEXT NOT NULL,
```

## 四、方向三：企业系统集成

### 4.1 目标价值

**解决的核心问题**：当前玄盾的告警仅通过 `tauri-plugin-notification`（`commands.rs` 第 110-114 行）推送桌面通知，企业运维场景下：
- 安全事件无法进入企业 IM 工作流（钉钉/飞书群），响应不及时
- 无法对接企业邮件系统，离线运维人员收不到告警
- 无法接入 SIEM/SOC 平台，玄盾成为安全监控盲区
- 无法通过 Webhook 触发自动化响应（如自动封禁 IP）

**交付价值**：
- 多通道告警确保安全事件及时触达运维人员
- SIEM 集成让玄盾融入企业既有安全运营体系
- Webhook 通道支持自动化响应编排
- 告警分级减少告警疲劳

### 4.2 功能详述

#### 4.2.1 钉钉机器人告警集成

- **配置项**：Webhook URL、Secret（加签模式）、@手机号列表、告警级别过滤
- **消息格式**：使用钉钉 Markdown 消息卡片，含攻击摘要、信任等级、时间、查看详情链接
- **告警分级**：
  - 紧急（拦截到高危攻击，如越狱成功迹象）→ @全体
  - 重要（拦截量突增 3 倍）→ @指定人员
  - 普通（常规拦截）→ 仅发送不 @
- **频率控制**：同类告警 5 分钟内合并，避免刷屏

#### 4.2.2 飞书机器人告警集成

- **配置项**：Webhook URL、签名密钥、@用户 open_id 列表、告警级别过滤
- **消息格式**：使用飞书交互式卡片消息，支持"查看详情""确认处理"按钮
- **告警分级**：同钉钉
- **频率控制**：同钉钉

#### 4.2.3 邮件告警（SMTP）

- **配置项**：SMTP 服务器、端口、用户名、密码、发件人、收件人列表、TLS/SSL 开关
- **邮件内容**：HTML 格式，含攻击摘要表格、趋势小图、详情链接
- **告警分级**：
  - 紧急：标题前缀 [紧急]
  - 重要：标题前缀 [重要]
  - 普通：标题前缀 [通知]
- **频率控制**：同类告警 10 分钟内合并为摘要邮件

#### 4.2.4 Webhook 通用告警

- **配置项**：Webhook URL、自定义请求头（如 Authorization）、请求体模板（支持变量替换）、重试策略
- **请求格式**：JSON，字段包括 `event_type`、`severity`、`timestamp`、`attack_category`、`text_preview`、`trust_level`、`reject_stage`、`engine_mode`
- **模板变量**：支持 `${attack_category}`、`${trust_level}`、`${timestamp}` 等变量
- **重试策略**：失败后指数退避重试 3 次（1s/4s/16s）

#### 4.2.5 SIEM 系统对接（Syslog）

- **协议支持**：RFC 5424（标准 Syslog）over UDP/TCP
- **配置项**：Syslog 服务器地址、端口、协议（UDP/TCP）、Facility（如 `local0`）、Severity 映射
- **消息格式**：结构化数据（SD-ID）携带玄盾特有字段
  ```
  <134>1 2026-07-11T14:32:21Z host xuandun 1234 - [xuandun@daoti event="block" category="jailbreak" trust="LOW" stage="reject_gate"] 检测到越狱攻击
  ```
- **Severity 映射**：
  - 紧急 → Critical (2)
  - 重要 → Warning (4)
  - 普通 → Informational (6)

#### 4.3.1 告警通道抽象层

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Optional

@dataclass
class AlertEvent:
    event_type: str          # "block" | "mode_switch" | "learning_complete" | "burst"
    severity: str            # "critical" | "important" | "info"
    timestamp: str
    attack_category: Optional[str]
    trust_level: str
    reject_stage: Optional[str]
    text_preview: str
    engine_mode: str
    extra: dict

class BaseNotifier(ABC):
    def __init__(self, config: dict):
        self.config = config

    @abstractmethod
    def send(self, event: AlertEvent) -> bool: ...

    @abstractmethod
    def validate_config(self) -> bool: ...

class DingTalkNotifier(BaseNotifier): ...
class FeishuNotifier(BaseNotifier): ...
class EmailNotifier(BaseNotifier): ...
class WebhookNotifier(BaseNotifier): ...
class SyslogNotifier(BaseNotifier): ...
```

#### 4.3.2 告警调度与去重

**新增文件**：`src/daoti_xuandun/integrations/alert_manager.py`

```python
class AlertManager:
    """告警管理器：负责分发、去重、频率控制。"""

    def __init__(self, notifiers: list[BaseNotifier]):
        self.notifiers = notifiers
        self._dedup_cache = {}  # key -> last_send_time

    def dispatch(self, event: AlertEvent):
        # 1. 去重判断（同类事件在冷却期内跳过）
        if self._is_duplicated(event):
            return
        # 2. 分级过滤（低级别不发送到紧急通道）
        for notifier in self.notifiers:
            if self._should_send(notifier, event):
                try:
                    notifier.send(event)
                except Exception:
                    logger.exception("Notifier %s failed", type(notifier).__name__)

    def _is_duplicated(self, event) -> bool: ...
```

### 6.3 数据库迁移说明

方向一和方向二均涉及 SQLite schema 变更，遵循现有迁移模式（参考 `db.rs` 第 78-112 行的 `hash_version` 迁移）：

1. 使用 `PRAGMA table_info` 检测字段是否存在
2. 缺失则 `ALTER TABLE ADD COLUMN`（SQLite 支持增量加列）
3. 新表使用 `CREATE TABLE IF NOT EXISTS`
4. 通过 `PRAGMA user_version` 管理迁移版本号

所有迁移均保证向前兼容，旧版本数据库可直接升级，无需数据导出/导入。
