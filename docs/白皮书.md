## 摘要

传统AI安全防御高度依赖攻击样本库——收集已知的提示注入、越狱、编码攻击等模式，构建黑名单或训练分类器。然而攻击技术快速迭代，新型变种层出不穷，基于攻击样本的防御永远落后一步。

道体·玄盾采用截然不同的技术路线：**以正常业务交互为主要学习素材，建立"正常域"的高维表征，偏离该域的行为即视为异常**。本白皮书详细阐述其背后的异常检测模型、多模态特征融合（语义特征+二进制特征+行为序列）以及活性自适应机制，解释为什么玄盾能够在无需预先收集攻击样本的前提下，对偏离正常域的输入进行检测。

**核心结论**：不需要攻击样本，不是因为系统"知道"攻击长什么样，而是因为它"知道"正常应该是什么样。这正是活性防护与传统安全产品的根本区别。

## 问题背景：攻击样本依赖的困境

当前主流的LLM安全防御方案可分为三类：

| 方案类型 | 工作原理 | 核心局限 |
|---------|----------|----------|
| **关键词黑名单** | 匹配预设的敏感词/短语 | 易被变体绕过（如"1gn0re"） |
| **对抗训练** | 用攻击样本微调模型 | 依赖已知攻击库，对未知变体泛化差 |
| **LLM裁判** | 用另一个LLM判断输入是否恶意 | 成本高、延迟大、可能被二次注入 |

这三类方案的共同特点是：**必须预先收集大量攻击样本**。攻击样本的收集本身即是难题——新攻击出现到被收录入库之间存在时间窗口（零日漏洞期），且攻击者可以持续制造变体。

玄盾从根本上改变了这一范式：**不再问"这是不是攻击"，而是问"这是不是正常"**。

---

## 核心理念：从"检测攻击"转向"检测异常"

设系统已学习到正常业务输入的分布 \( P_{normal}(x) \)。对于新输入 \( x_{new} \)，计算其与正常分布的偏离程度：

\[
\text{anomaly\_score} = d(x_{new}, D_{normal})
\]

其中 \( D_{normal} \) 是正常域的紧凑表征（原型向量集合），\( d \) 是距离度量（如余弦距离、马氏距离）。当偏离程度超过阈值时，判定为异常（潜在攻击）。

该范式在网络安全领域已被证明有效（如基于流量的异常检测），但在AI安全领域长期未得到应用，主要受制于两个技术瓶颈：
- **高维语义空间的表征**：如何将任意自然语言输入映射到可度量的数学空间？
- **动态域演化**：正常业务模式会随时间变化（如客服系统增加新产品），如何持续更新正常域？

玄盾通过**洛书符号映射器**和**在线原型演化**解决了这两个瓶颈。

### 整体架构

```
输入（任意语言/格式）
    ↓
[洛书符号映射器] → 高维符号向量（洛书空间）
    ↓
[内生域感知] → 计算与安全原型的距离 + 局部密度
    ↓
[多模态异常检测] → 结构异常 + 二进制异常 + 时序行为
    ↓
[活性决策引擎] → 允许/拒绝/混沌期孵化
    ↓
[拦截响应] → 通知 + 日志 + 审计记录
```

## 洛书符号映射器：语言无关的通用表征

### 设计哲学

符号级防护的核心诉求是：**不依赖任何特定语言的先验知识（如英文语法、中文分词）**，将任何输入——无论自然语言、编码载荷、二进制数据——统一映射到同一个高维数学空间。

玄盾的洛书映射器受华夏哲学中的"阴阳"与"八卦"启发：所有信息可被分解为阴阳二元的动态递归流。工程实现上，通过以下步骤将输入编码为洛书空间向量：

1. **字节级散列**：将输入UTF-8字节流通过位置敏感哈希映射到高维稀疏向量。
2. **阴阳分叉**：根据输入熵值动态调整阴阳分量占比（高熵输入→阳增强，低熵输入→阴增强）。
3. **穿透门控**：通过自适应门控网络融合阴阳分量，产生最终状态向量。
4. **正交投影**：通过随机正交投影矩阵将洛书空间无损映射到玄盾内部维度。

### 为何语言无关

- 输入仅被当作字节流处理，不进行分词、词性标注、语法解析。
- 散列函数对所有Unicode字符一视同仁，不区分拉丁、西里尔、汉字等。
- 熵值计算基于字节频率，不依赖语义。

**设计目标**：同一攻击指令的中文、英文、甚至Base64编码版本，在洛书空间中理论上会落在相近区域（攻击原型附近）；而正常业务请求（无论何种语言）会落在安全原型附近。实际效果取决于输入长度、编码方式和原型库覆盖度。

> 洛书映射器的核心参数（随机投影矩阵、阴阳分叉阈值等）在初始化后即固定，运行时不再调整，确保映射的确定性和可重现性。详细算法原理参见附录A。

## 内生域感知：在线原型演化与冷启动策略

### 原型库结构

玄盾维护两个原型库：
- **安全原型库**：存储正常业务输入的洛书向量聚类中心（每个原型代表一类正常模式）。
- **攻击原型库**：存储已被判定为攻击的输入向量（用于**二次加速**，非必要检测条件）。

初始时，安全原型库可通过用户提供的少量正常样本预热（可选），或从空库开始在线学习。

### 在线学习机制

对于每个被允许的输入，系统执行：

1. 计算输入向量与所有安全原型的距离，找到最近原型。
2. 若距离小于阈值 → 将该原型向输入方向轻微移动（学习率动态调整）。
3. 若距离大于阈值 → 创建新原型（代表新的正常模式）。

这一机制使得系统**从正常运行中持续学习**，无需人工标注攻击样本。随着业务演化（如新增产品、新话术），系统自动扩展正常域表征。

### 混沌期孵化（新域发现）

当输入与所有安全原型的距离都较大，且未检测到明确攻击特征时，系统不会立即拒绝，而是：
- 将其放入"混沌期候选队列"。
- 若同一模式反复出现且始终未触发攻击信号，则自动晋升为新安全原型。

### 冷启动安全策略

系统部署后的初始阶段（安全原型数 < 100），启用三重保护：

1. **内置通用安全原型库**：预置少量通用文本原型（涵盖中英文日常对话、代码片段、SQL 查询等），确保首日即有基础防御能力。该原型库不包含任何业务敏感信息，仅提供通用的"正常文本"分布参考。

2. **高敏感初始阈值**：冷启动期间，异常检测阈值较正常运行调低 30%，优先保证拦截率（宁可误报，不可漏报）。所有拦截结果进入人工复核队列（通过管理界面），管理员可将误报标记为"安全"并自动加入安全原型库。

3. **强制预热机制**：若客户能提供至少 100 条正常业务对话（推荐在部署前收集），系统可绕过冷启动阶段，直接进入稳定运行模式。

冷启动期通常在 1000 次正常交互后结束，此时系统已建立稳定的域分布。

### 攻击原型库的更新策略（性能优化层）

当系统判定某输入为攻击后：
- 该输入的洛书向量被加入攻击原型库（经过去重和聚类合并）。
- 后续遇到与已记录攻击距离极近的输入，可直接快速拒绝，避免重复计算。
- 为防止污染，攻击原型库采用**频率门限**：同一模式需被检测到至少 3 次才被固化；且攻击原型不参与正常域的距离比较（仅作为独立加速索引）。

因此，系统不依赖攻击原型库来发现新攻击（那是异常检测的工作），攻击原型库只是性能优化层。

## 多模态异常检测：语义、二进制、时序融合

仅依靠洛书距离不足以区分所有攻击类型（尤其短文本）。玄盾融合三个维度的异常信号，形成综合判决。

### 结构异常（语义层）

**目的**：捕获命令式语法、非常规排版、异常标点密度等与正常业务不一致的文本结构。

**技术实现**：
- 统计大写字母比例、感叹号/问号密度、特殊字符比例。
- 检测命令式动词模式，但这些模式是从正常域数据中**统计学习**的，非硬编码黑名单。
- 古典中文偏离度检测（用于现代中文业务场景）。

### 二进制异常（字节层）

**目的**：检测编码攻击（Base64/Hex）、混淆攻击（leet speak、同形符）、控制字符注入等。

**技术实现**：
- **熵值检测**：正常文本的字节熵通常在3.5-4.5之间，编码攻击的熵可能接近8（完全随机）或低于2（高度重复）。
- **重复模式检测**：检测连续相同字节或短周期重复。
- **编码特征检测**：识别Base64字符集占比、Hex字符分布、Unicode同形符混用。
- **控制字符比例**：检测换行、制表、空字节等异常比例。

### 行为序列（时序层）

**目的**：捕获多轮对话中的重复攻击模式、低频试探、渐进式越狱。

**技术实现**：
- 维护每个会话（session）的符号序列历史。
- 计算当前符号序列与历史窗口的**马氏距离**，使用异常剔除（MAD）防止窗口污染。
- 检测短时间内的重复请求模式（同一攻击变体多次提交）。

### 多模态融合判决

三个维度的异常得分定义如下：
- `s_struct`：结构异常得分（0-1，越高越异常）
- `s_binary`：二进制异常得分（0-1）
- `s_temporal`：时序行为异常得分（0-1，仅在有足够历史时启用）

**融合公式**：
```
final_score = w1 * s_struct + w2 * s_binary + w3 * s_temporal
```
权重 `(w1, w2, w3)` 根据安全模式配置：

| 模式 | w1（结构） | w2（二进制） | w3（时序） | 适用场景 |
|------|-----------|-------------|-----------|----------|
| 均衡（默认） | 0.4 | 0.4 | 0.2 | 通用业务 |
| 高安全 | 0.3 | 0.5 | 0.2 | 金融、医疗 |
| 低误报 | 0.5 | 0.3 | 0.2 | 客服、教育 |

最终判决：若 `final_score > θ_decision` 则拒绝，否则允许。`θ_decision` 默认 0.6，可通过管理界面动态调整。

> 当 `s_temporal` 因历史不足不可用时，其权重按比例分配给 `s_struct` 和 `s_binary`。

## 活性自适应：让系统学会"什么时候该用语言特征"

玄盾的活性机制：**随着正常域原型的积累，系统自动降低对语言特征的依赖，最终实现纯符号级决策**。

### 语言特征权重渐进衰减

衰减函数基于原型覆盖率和运行时间的指数平滑曲线：

```
lang_weight = max(0, 1 - (safe_n / safe_max)^α - (attack_n / attack_max)^β)
```

其中：
- `safe_max = 1000`, `attack_max = 500`（饱和上限）
- α = 0.5, β = 0.3（控制衰减速度，基于多组业务日志模拟确定）
- 同时引入时间因子：运行满 30 天后，`lang_weight` 额外乘以 0.9；满 90 天后乘以 0.8。

**实际效果**：
- 小型业务（每天几十次交互）即使在运行一年后仍会保留部分语言特征（约0.3-0.5），这是合理的——因为原型库不足，需要语言特征辅助。
- 大型业务（每天数千次交互）可能在数周内将语言特征权重降至接近0，实现纯符号防护。

### 洛书置信度豁免

对于洛书信号已足够强的输入（安全距离远低于阈值 或 攻击距离显著偏高），系统直接忽略语言特征，避免误判。该机制基于数据驱动，无需人工配置阈值。

## 预处理管道：增强编码与混淆攻击检测

为进一步增强对编码混淆攻击的防御，玄盾提供可选预处理管道（默认关闭，可通过配置开启）：

- **Base64/Hex解码**：尝试解码输入，若解码后的文本被判定为攻击，则拒绝原始输入。
- **Unicode正规化**：将同形符（如西里尔字母伪装拉丁字母）转换为标准形式，再进行检测。
- **命令式短语白名单**：内置技术领域常见命令式短语（如"Write a SQL query"、"Explain how..."），降低技术性良性文本的误判率。

这些预处理步骤同样不需要攻击样本——解码后的文本与正常业务的分布对比，即可判断是否为攻击。

玄盾桌面端应用将核心防护引擎从服务端扩展到桌面，为所有运行 AI Agent 的桌面电脑提供实时安全守护。

### 产品定位

> 让每一台运行 AI Agent 的桌面电脑，都有一道看不见的盾。

当前 AI Agent 生态的关键趋势：
- Cursor、Windsurf、Claude Desktop 等 IDE/Agent 将 AI 能力直接嵌入桌面
- Agent 系统拥有文件系统、终端、网络等本地资源访问权限
- 恶意提示注入在桌面端的危害远超云端（可执行任意命令）
- **目前鲜见桌面级产品为 Agent 系统提供实时安全防护**

## 常见技术质疑与回应（FAQ）

### Q1：如果业务本身包含高熵数据（如用户粘贴代码片段），二进制异常检测会不会大量误报？

**A**：会，但可通过以下方式解决：
- 系统内置代码模式识别预处理，可检测 Markdown 代码块、缩进模式、括号配对和编程关键词密度，将高熵但结构化的代码内容标记为"已知模式"，降低异常得分。
- 管理员可将代码粘贴类输入加入 `warmup_safe` 或通过管理界面标记为"安全"，系统自动学习该模式。
- 若业务中代码片段常见，推荐使用"低误报模式"。

### Q2：洛书空间为什么是特定维度？有消融实验支撑吗？

**A**：洛书空间的维度源于道体内部向量检索引擎的原型向量维度，经过了多轮内部消融实验（64/128/176/256 对比），在精度-效率平衡上最优。该维度在多个业务中验证了有效性。如需详细技术报告，可联系技术支持。

### Q3：无需攻击样本，如何保证对零日攻击的防御率？有没有理论下界？

**A**：在异常检测框架下，防御率取决于正常域的覆盖度。理论上，若攻击输入与所有正常样本的统计分布差异可被检测（差异效应量 > δ），则防御率可达到 1-ε。实践中，我们通过持续学习扩展正常域，使攻击表面越来越小。无法给出绝对下界，但通过混沌期孵化机制，漏过的攻击会在多次出现后被捕获。

### Q4：原型库会无限增长吗？是否有合并或遗忘机制？

**A**：有。原型库上限默认为 512，超过后触发合并（余弦相似度 >0.85 的原型合并）和遗忘（低频原型淘汰），保证内存可控。管理员可调整上限。

### Q5：活性衰减后，如果业务模式发生剧变，系统多久能重新适应？

**A**：业务剧变会表现为大量输入与现有原型距离增大，触发混沌期孵化。系统会快速创建新原型，通常 50-100 次交互后即可稳定。若剧变是永久性的，建议管理员执行"重置原型库"并重新预热，或导入新的预热样本。

### Q6：你们声称"无需攻击样本"，那为什么附录C中提供了调优案例？

**A**：预热样本（`warmup_safe`）是正常业务样本，不是攻击样本。附录C中的调优案例主要是为了处理**正常业务中的边缘情况**（如技术术语、代码片段），而非教客户如何收集攻击样本。两者的本质区别在于：攻击样本需要安全专家从海量威胁情报中收集，而正常样本客户每天都有。

### Q7：基准测试取得了 100% 的成绩，这是否意味着系统无漏洞？

**A**：不是。100% 的成绩仅表示系统已完整覆盖本报告三项测试套件中的攻击模式。这些测试套件在开发期间被用于回归验证，检测规则经过多轮针对性调优，因此取得满分是预期的。但这**不代表对未知/新型攻击的绝对防御能力**——真实对抗环境中的表现会低于本报告数据。符号级检测存在理论边界：当良性样本与攻击样本在字节分布上高度重叠时，系统无法区分。建议使用独立第三方测试集或红队对抗评估来获得客观的实战能力数据。系统通过 `--feedback` 机制持续在线学习，漏检样本可回灌进化防御能力。

## 附录A：洛书映射器核心算法原理

### A.1 字节级散列

给定输入字节流，通过位置敏感哈希将每个字节映射到高维空间，并引入位置调制以保留顺序信息。具体实现涉及固定随机投影矩阵和正弦位置权重，确保：
- 相同字节在不同位置产生不同贡献
- 映射过程完全确定性（无随机性）
- 对任意长度的输入均可处理

### A.2 阴阳分叉

计算输入字节熵值，根据熵值动态分配阴阳权重：
- 低熵输入（如重复文本）偏阴（稳定态）
- 高熵输入（如编码载荷）偏阳（活跃态）

阴阳分量分别通过独立的线性变换后加权融合，产生状态向量。

### A.3 穿透门控

门控网络根据状态向量自适应调节信息流通量。`p_proto` 是输入向量与当前所有安全原型的余弦相似度经 softmax 归一化后的概率分布。门控训练目标为最大化原型库的分离度，鼓励输入均匀分布到各原型，避免坍缩。

### A.4 洛书空间向量的最终形成

状态向量通过固定的正交投影矩阵映射到洛书空间，该投影矩阵在初始化时生成并固定。最终输出的洛书向量是单位向量（L2范数归一化），确保距离度量的稳定性。

> *洛书映射器的完整实现细节（包括矩阵维度、激活函数、训练方法）受商业秘密保护。如需更详细的技术文档，请在签署NDA后联系技术团队。*

## 附录B：洛书 vs 传统 Embedding 消融实验（内部对比）

> 以下为内部测试的定性对比结论。所有对比均在相同测试集和相同决策逻辑下进行，仅替换输入向量编码方式。

| 对比维度 | Sentence-BERT (中文) | OpenAI text-embedding-3 (API) | 洛书符号映射器 |
|---------|---------------------|-------------------------------|--------------|
| 语言依赖 | 仅中文 | 多语言（但有过滤） | **无** |
| 编码载荷处理 | 效果有限（语义丢失） | 效果有限（语义丢失） | **原生支持** |
| 延迟量级 | ~15ms | ~45ms（含API调用） | **<1ms** |
| 需要外部服务 | 否 | 是（API调用） | **否** |

**核心结论**：
1. **语言无关性**：洛书映射器不依赖任何特定语言的先验知识，可处理任意字节流（包括编码载荷、二进制数据）。相比依赖分词或语义模型的方案，其在多语言混合和编码攻击场景下有结构性优势。
2. **延迟优势**：洛书映射器为纯本地计算，无API调用开销，延迟比传统embedding低一个数量级。
3. **局限性**：洛书映射器不包含语义信息，对语义高度重叠的短文本区分能力弱于语义embedding。
