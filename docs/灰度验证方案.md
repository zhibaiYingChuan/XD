# 道体·玄盾 灰度验证方案（v1.2.0）

> **目标**：在 1-2 个内部环境部署运行一周，验证 Dashboard 可视化、安全报告推送、企业告警集成三大模块在真实流量下的表现。
> **适用版本**：v1.2.0
> **验证周期**：7 天（建议覆盖工作日 + 周末）

---

## 一、灰度环境规划

### 1.1 环境选择

| 环境 | 用途 | 流量特征 | 接入方式 |
|------|------|---------|---------|
| 环境 A（开发团队） | 日常开发使用 AI 编码工具 | 中等流量，工作日高峰 | 代理模式（Claude Desktop / Cursor） |
| 环境 B（安全测试团队） | 攻击模拟与防护验证 | 低流量，高攻击占比 | SDK 集成 + 模拟测试 |

### 1.2 部署清单

每个灰度环境需要：

- [ ] 玄盾桌面端 v1.2.0 已安装并启动
- [ ] Python 引擎进程运行正常（/health 返回 200）
- [ ] AI 工具已通过代理或 SDK 接入玄盾
- [ ] 告警通道至少配置 1 个（推荐 Webhook，便于测试）
- [ ] 灰度监控脚本已部署（scripts/graymon.py）

### 1.3 隔离策略

- 灰度环境的玄盾实例使用独立的 SQLite 数据库（默认隔离）
- 告警通道使用专用测试 Webhook，不发送到生产告警群
- 报告生成功能在灰度环境独立运行，不影响生产数据

## 二、验证指标与检查点

### 2.1 核心指标

| 指标 | 验证方法 | 预期结果 | 告警阈值 |
|------|---------|---------|---------|
| 引擎可用性 | 监控脚本每 60s 探测 /health | 7天内可用率 ≥ 99.5% | 连续 3 次探测失败 |
| 拦截响应延迟 | /protect 接口 latency_ms | P95 < 200ms | P95 > 500ms |
| Dashboard 趋势图加载 | 手动切换 1H/24H/7D/30D | 1秒内渲染完成 | 加载 > 3秒 |
| 报告生成时间 | 生成周报 | < 10秒 | > 30秒 |
| 告警通道连通性 | 测试告警按钮 | 5秒内收到通知 | 发送失败 |

### 2.2 每日检查点

**Day 1-2：基础接入验证**
- [ ] AI 工具流量已被玄盾检测（Dashboard total_requests > 0）
- [ ] 观察模式正常学习（sample_count 持续增长）
- [ ] 自动切换到保护模式触发（sample_count >= 1000）
- [ ] Dashboard 趋势图显示实时数据

**Day 3-4：防护效果验证**
- [ ] 运行模拟测试，拦截率 ≥ 90%
- [ ] 真实攻击流量被拦截（查看日志页面）
- [ ] 误报率可接受（良性请求被拦截 < 5%）
- [ ] 告警通道在拦截时正确触发

**Day 5-6：报告与可视化验证**
- [ ] 生成日报/周报，内容完整
- [ ] 攻击分类饼图数据准确
- [ ] 周度对比柱状图显示趋势
- [ ] 实时监控指标卡片更新正常

**Day 7：综合评估**
- [ ] 收集运维人员反馈
- [ ] 检查 7 天趋势数据完整性
- [ ] 验证告警去重逻辑（同类攻击 5 分钟内不重复告警）
- [ ] 确认报告归档功能正常

### 2.3 告警通道专项验证

| 通道 | 验证步骤 | 常见问题 |
|------|---------|---------|
| 钉钉 | 配置机器人 webhook + 加签 secret | 签名错误：检查 timestamp 毫秒级 |
| 飞书 | 配置自定义机器人 webhook + 签名 | IP 白名单：需将玄盾服务器 IP 加入飞书机器人配置 |
| 邮件 | 配置 SMTP 服务器 + 认证 | 防火墙拦截：确认 25/465/587 端口可用 |
| Webhook | 配置目标 URL + 自定义 headers | HTTPS 证书：玄盾使用 urllib，需目标 URL 证书有效 |
| Syslog | 配置 syslog 服务器 host:port | UDP 丢包：高告警量时建议使用 TCP |

## 三、监控脚本使用

### 3.1 部署监控脚本

```bash
# 在灰度环境机器上运行
python scripts/graymon.py --engine http://127.0.0.1:18770 --interval 60 --output /var/log/xuandun-graymon.jsonl
```

### 3.2 参数说明

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--engine` | http://127.0.0.1:18770 | 玄盾引擎地址 |
| `--interval` | 60 | 探测间隔（秒） |
| `--output` | stdout | 日志输出路径（JSONL 格式） |
| `--alert-webhook` | 无 | 异常告警 Webhook URL |
| `--latency-threshold` | 500 | 延迟告警阈值（ms） |

### 3.3 监控指标

脚本每 `--interval` 秒采集以下指标并输出 JSONL 日志：

```json
{
  "timestamp": "2026-07-11T10:00:00Z",
  "engine_health": "ok",
  "engine_uptime": 3600.5,
  "total_requests": 1523,
  "total_blocked": 87,
  "block_rate": 0.0571,
  "latency_p50_ms": 45.2,
  "latency_p95_ms": 128.7,
  "latency_p99_ms": 245.1,
  "learning_mode": "protecting",
  "sample_count": 1523,
  "alert": null
}
```

当指标超过阈值时，`alert` 字段不为 null，同时发送到 `--alert-webhook`。

### 3.4 分析日志

```bash
# 统计可用率
python scripts/graymon.py --analyze /var/log/xuandun-graymon.jsonl --metric availability

# 查看延迟分布
python scripts/graymon.py --analyze /var/log/xuandun-graymon.jsonl --metric latency

# 导出 7 天汇总报告
python scripts/graymon.py --analyze /var/log/xuandun-graymon.jsonl --summary --output graymon-report.json
```

## 四、回滚方案

### 4.1 回滚触发条件

满足以下任一条件立即回滚：

1. 引擎连续崩溃（1小时内崩溃 ≥ 3 次）
2. 拦截延迟 P95 > 1000ms 持续 30 分钟以上
3. 误报率 > 20%（良性请求被拦截比例）
4. 告警通道持续发送失败（24小时内成功率 < 50%）
5. Dashboard 数据查询导致应用卡死或崩溃

### 4.2 回滚步骤

**步骤 1：停止灰度实例**

```bash
# 停止引擎进程
# Windows
taskkill /F /IM xuandun-desktop.exe
# Linux/macOS
pkill -f xuandun-desktop
```

**步骤 2：恢复 AI 工具直连**

- 代理模式：移除 AI 工具中的代理配置（127.0.0.1:18765）
- SDK 模式：在应用代码中注释或移除 `shield.protect()` 调用

**步骤 3：备份灰度数据**

```bash
# 备份灰度环境的数据库（用于事后分析）
copy %APPDATA%\com.xuandun.desktop\xuandun.db xuandun-gray-backup.db
```

**步骤 4：降级到 v1.1.0（如需）**

```bash
# 卸载 v1.2.0
# Windows: 控制面板 → 卸载程序 → 道体·玄盾
# 安装 v1.1.0
XuanDun_1.1.0_x64-setup.exe /S
```

**步骤 5：记录回滚原因**

填写回滚报告，包含：
- 回滚触发条件
- 问题复现步骤
- 相关日志摘录（graymon.jsonl 片段）
- 影响范围评估

### 4.3 灰度数据保留

回滚后灰度环境的数据（SQLite 数据库）应保留至少 30 天，用于：
- 问题根因分析
- 后续版本修复验证
- 基准对比数据

## 五、灰度验收标准

灰度验证 7 天后，满足以下全部条件即可通过验收，进入正式发布：

| 验收项 | 标准 | 验证方法 |
|--------|------|---------|
| 引擎稳定性 | 7天可用率 ≥ 99.5% | graymon 日志统计 |
| 拦截性能 | P95 延迟 < 200ms | graymon 延迟分析 |
| Dashboard 可用性 | 无加载失败或卡死 | 运维人员确认 |
| 报告生成 | 周报成功生成并归档 | Reports 页面检查 |
| 告警通道 | 至少 1 通道 7天内稳定推送 | 告警日志检查 |
| 误报率 | < 5% | 日志页面人工审核 |
| 运维反馈 | 无阻塞性问题 | 运维人员签字确认 |

---

## 六、灰度环境清理

验收通过后：

1. 停止灰度环境监控脚本
2. 导出灰度数据汇总报告（归档到 docs/graymon-report-*.json）
3. 卸载灰度环境玄盾实例
4. 恢复 AI 工具到生产配置
5. 将灰度经验纳入正式部署指南
